## 1. 개요

### 1.1 제품 개요
PMS 는 IT 기업의 프로젝트 관리 효율화를 위한 웹 및 모바일 기반 프로젝트 관리 시스템입니다. PM 은 프로젝트 정보 입력 및 일정 관리를, 관리자는 프로젝트 검토 및 모니터링을 수행할 수 있습니다.

### 1.2 목표
- 프로젝트 정보의 표준화된 입력 및 관리
- 시각적 일정 관리 (간트 차트) 를 통한 진행 상황 투명성 확보
- 관리자 중심의 감시 및 승인 프로세스 구현
- PC 및 모바일 환경 모두에서 접근 가능한 유연성 제공

### 1.3 대상 사용자
| 사용자 역할 | 설명 | 주요 권한 |
|------------|------|----------|
| PM (Project Manager) | 일반 사용자 | 프로젝트 생성/수정/조회, 일정 관리 |
| Admin (임원/PMO) | 관리자 | 전체 프로젝트 조회, 승인/거절, 모니터링, 리포트 출력 |

## 3. 사용자 스토리 (User Stories)

### 3.1 PM 사용자 스토리

```
US-001: 프로젝트 정보 입력
 作为: PM
 我想: 프로젝트 정보를 입력할 수 있도록
 所以当: 새로운 프로젝트가 시작될 때
  나는: 프로젝트명, 코드, 고객사, 담당자, 일정을 입력하고 제출할 수 있어야 한다

US-002: 프로젝트 일정 관리
 作为: PM
 我想: 간트 차트를 통해 일정을 시각적으로 관리할 수 있도록
所以当: 프로젝트 일정을 조정해야 할 때
  나는: 드래그 앤 드롭으로 일정을 변경하고 변경 내역을 저장할 수 있어야 한다

US-003: 프로젝트 조회
 作为: PM
 我想: 내가 등록한 프로젝트 목록을 조회할 수 있도록
所以当: 현재 진행 중인 프로젝트를 확인해야 할 때
  나는: 프로젝트 상태, 일정, 진행률을 확인할 수 있어야 한다

US-004: 모바일 접근
 作为: PM
 我想: 모바일에서도 프로젝트 정보를 조회하고 업데이트할 수 있도록
所以当: 외부 현장에서 프로젝트 상태를 확인해야 할 때
  나는: 모바일 브라우저에서 시스템에 접근하고 주요 작업을 수행할 수 있어야 한다
```

### 3.2 관리자 스토리

```
US-101: 전체 프로젝트 모니터링
 作为: 관리자 (임원/PMO)
 我想: 모든 프로젝트의 현황을 한눈에 확인할 수 있도록
所以当: 프로젝트 진행 상황을 감독해야 할 때
  나는: 프로젝트 목록, 상태, 위험 요소를 대시보드에서 볼 수 있어야 한다

US-102: 프로젝트 승인
 作为: 관리자 (PMO)
 我想: 제출된 프로젝트 일정을 검토하고 승인/거절할 수 있도록
所以当: PM 이 제출한 프로젝트 계획을 검토해야 할 때
  나는: 승인 또는 거절 사유를 입력하고 통보할 수 있어야 한다

US-103: 리포트 출력
 作为: 관리자 (임원)
 我想: 프로젝트 현황 리포트를 출력할 수 있도록
所以当: 임원 회의로 보고 자료를 준비해야 할 때
  나는: PDF 또는 Excel 형식으로 리포트를 다운로드할 수 있어야 한다
```

### 3.3 시스템 스토리

```
US-SYS-001: 보안 인증
 作为: 시스템
 我想: 모든 요청에 대한 인증을 수행하도록
所以当: 사용자가 API 를 호출할 때
  나는: JWT 토큰을 검증하고 권한이 있으면 요청을 처리해야 한다

US-SYS-002: 데이터 무결성
 作为: 시스템
 我想: 프로젝트 데이터의 일관성을 유지하도록
所以当: 여러 사용자가 동시에 데이터를 수정할 때
  나는: 트랜잭션 관리로 데이터 불일치를 방지해야 한다
```

## 4. 비기능 요구사항 (Non-Functional Requirements)

| 요구사항 | 목표치 | 측정 방법 |
|---------|-------|----------|
| 페이지 로드 시간 | 2 초 이내 | 페이지 로딩 시간 측정 |
| API 응답 시간 | 500ms 이내 (95% 성능) | API 응답 시간 모니터링 |
| 동시 사용자 지원 | 100 명 이상 | 부하 테스트 |

| 요구사항 | 설명 |
|---------|------|
| 인증 | JWT 토큰 기반 인증, 비밀번호hashed 저장 (bcrypt) |

| 입력 검증 | 모든 입력 값 sanitization, SQL injection 방지 |
| API 보안 | Rate limiting, CORS 정책 설정 |

### 4.3 확장성 (Scalability)

| 요구사항 | 설명 |
|---------|------|

## 7. 데이터 모델 (Data Model)

### 7.1 주요 엔티티

```
User
├── id: UUID (PK)
├── email: VARCHAR(255) UNIQUE
├── password_hash: VARCHAR(255)
├── name: VARCHAR(100)
├── role: ENUM('PM', 'ADMIN')
├── is_active: BOOLEAN
├── created_at: TIMESTAMP
└── updated_at: TIMESTAMP

Project
├── id: UUID (PK)
├── project_code: VARCHAR(50) UNIQUE
├── project_name: VARCHAR(255)
├── client_name: VARCHAR(255)
├── designer: VARCHAR(100)
├── developer: VARCHAR(100)
├── start_date: DATE
├── end_date: DATE
├── status: ENUM('DRAFT', 'SUBMITTED', 'APPROVED', 'REJECTED', 'IN_PROGRESS', 'COMPLETED')
├── approval_status: ENUM('PENDING', 'APPROVED', 'REJECTED')
├── rejection_reason: TEXT
├── created_by: UUID (FK → User)
├── created_at: TIMESTAMP
└── updated_at: TIMESTAMP

ProjectSchedule
├── id: UUID (PK)
├── project_id: UUID (FK → Project)
├── task_name: VARCHAR(255)
├── start_date: DATE
├── end_date: DATE
├── progress: INTEGER (0-100)
├── priority: ENUM('LOW', 'MEDIUM', 'HIGH', 'CRITICAL')
└── order: INTEGER
```

## 10. 테스트 전략 (Testing Strategy)

### 10.1 테스트 계층

| 계층 | 테스트 타입 | 도구 | 목표 커버리지 |
|------|-----------|------|-------------|
| Unit | 단위 테스트 | pytest | 80% 이상 |
| Integration | 통합 테스트 | pytest-asyncio | 핵심 흐름 100% |
| E2E | 엔드 투 엔드 | Playwright | 주요 시나리오 100% |
| Performance | 성능 테스트 | locust | 부하 조건 검증 |
| Security | 보안 테스트 | bandit, safety | 취약점 검사 |

### 10.2 주요 테스트 시나리오

1. **사용자 인증**
   - 유효한 자격증명으로 로그인 성공
   - 잘못된 비밀번호로 로그인 실패
   - 세션 만료 후 자동 로그아웃

2. **프로젝트 CRUD**
   - 프로젝트 생성 및 저장
   - 프로젝트 수정 및 업데이트
   - 프로젝트 조회 및 필터링
   - 프로젝트 삭제 (soft delete)

3. **승인 프로세스**
   - PM 이 프로젝트 제출
   - Admin 이 승인/거절
   - 상태 변경 알림

4. **간트 차트**
   - 일정 조회 및 표시
   - 일정 수정 (드래그 앤 드롭)
   - 일정 충돌 검증
